# Audio Processing Study

This document describes how audio is processed in the current `apps/api` implementation, from upload to transcription to RAG indexing.

## Scope

There are two separate audio paths in the backend:

1. Subject textbook audio ingestion through `apps/api/src/subjects/subject.controller.ts`
2. Direct speech transcription APIs in `apps/api/src/speech/speech.routes.ts`

The first path is the one that feeds the knowledge base. The second is an ad hoc transcription API for client-side recording or uploads.

## Main ingestion flow

Teacher uploads audio through:

- `POST /api/subjects/with-textbook`
- `POST /api/subjects/:id/textbooks`
- `POST /api/subjects/:id/textbooks/batch`

The controller detects audio files using `AUDIO_EXTENSIONS`, computes a SHA-256 hash with `computeFileHash()`, and creates or reuses an `AIJob` through `findOrCreateAIJob()` in `apps/api/src/video-jobs/videoProcessor.ts`.

Audio jobs are stored as:

- `AIJob.fileType = "AUDIO"`
- `AIJob.status = queued | processing | completed | failed`
- `AIJob.audioMetadata`
- `AIJob.transcript`
- `AIJob.transcriptSegments`
- `AIJob.chunksCreated`

The queue worker is started from `apps/api/src/index.ts` with `startQueueWorker(10000)`, so jobs are polled about every 10 seconds.

## Audio worker pipeline

Audio jobs are processed in `processAudioJob()` inside `apps/api/src/video-jobs/videoProcessor.ts`.

Pipeline:

1. Extract metadata with `getAudioMetadata()`
2. Transcribe with timestamps using `transcribeAudioFileWithTimestamps()`
3. Store formatted transcript in the job
4. If transcript text is long enough, embed the plain transcript text into `KnowledgeChunk`
5. Update the matching textbook record in `Subject.textbooks`
6. Start async summarization

The code treats transcripts shorter than 100 trimmed characters as "not enough to index". Those jobs still complete, but no embeddings are written.

## Transcription implementation

Core transcription logic lives in `apps/api/src/utils/transcribeAudio.ts`.

Important behavior:

- Required env vars: `WHISPER_API_BASE_URL`, `WHISPER_API_KEY`
- Default model: `qwen3-asr-1.7b`
- Supported input extensions: `.mp3`, `.wav`, `.m4a`, `.ogg`, `.flac`, `.aac`, `.mp4`
- `.mp4` is accepted here because the utility can extract audio from video first with `ffmpeg`

There are two transcription modes:

1. `transcribeAudioFile()`
   - Uses chunking when a file is over about 20 MB or longer than 13 minutes
   - Chunk duration target is 780 seconds
   - Merges chunk text and chunk segments

2. `transcribeAudioFileWithTimestamps()`
   - Splits audio into 30 second WAV chunks with `ffmpeg`
   - Transcribes each chunk separately
   - Creates coarse timestamp segments by assigning each chunk to a `[start, end]` window
   - This is the mode used by the background audio and video workers

Formatted transcript output is generated by `formatTranscriptWithTimestamps()`, which produces lines like:

```text
[00:00 - 00:30] transcript text
```

## Chunking and temporary files

Chunking logic is split across two helpers:

- `apps/api/src/utils/audioChunker.ts`
- `apps/api/src/utils/transcribeAudio.ts`

Behavior:

- Large file chunking uses `ffmpeg -f segment -c copy`
- Timestamp chunking uses `ffmpeg` re-encoding into 16 kHz mono WAV
- Temporary chunk files are cleaned up after processing
- If input was an `.mp4`, temporary extracted WAV audio is also cleaned up

Operational dependency:

- `ffmpeg` must be available on the server path

## Embedding and RAG indexing

After successful transcription, `processAudioJob()` indexes the raw transcript text with:

- `indexContent(subjectId, formLevel, transcriptionResult.text, "audio_transcript")`

`indexContent()` lives in `apps/api/src/utils/embeddings.ts` and:

1. Splits text into overlapping chunks
2. Generates embeddings sequentially
3. Stores chunks in `KnowledgeChunk`

Audio transcript chunks are saved with:

- `sourceType = "audio_transcript"`
- subject and form-level scoping

The formatted transcript with timestamps is stored on the textbook record, but the embedded text is the plain transcript content.

## Subject textbook updates

When a job finishes, `updateSubjectTextbook()` writes back to `Subject.textbooks[]`.

For audio, the main fields are:

- `transcriptionStatus`
- `transcriptText`
- `audioDuration`
- `transcriptUrl`
- `transcriptFileName`

The worker also writes a transcript text file into `UPLOADS_DIR` or `process.cwd()/uploads`.

## Summary generation

After indexing, `fireAndForgetSummarize()` runs asynchronously and writes textbook summary state into:

- `textbooks.$.summary.content`
- `textbooks.$.summary.generatedAt`
- `textbooks.$.summary.status`

The summary source text is the full transcript text, not the timestamp-formatted version.

## Direct speech API

`apps/api/src/speech/speech.routes.ts` exposes:

- `POST /api/speech/transcribe`
- `POST /api/speech/transcribe-base64`
- `GET /api/speech/status`

This path is separate from AI jobs.

Behavior:

- Uses multer memory upload, up to 200 MB
- Tries OpenAI-compatible `/audio/transcriptions` first
- Falls back to `/chat/completions` with `audio_url` content
- Returns transcription immediately in the HTTP response
- Does not create `KnowledgeChunk`
- Does not update `Subject` or `AIJob`

It uses a different config set:

- `ASR_API_BASE`
- `ASR_MODEL`
- `OPENAI_API_KEY`

This is operationally distinct from the background worker path, which uses `WHISPER_*`.

## Data model impact

Audio processing touches two main collections:

1. `AIJob`
2. `KnowledgeChunk`

And one embedded document structure:

3. `Subject.textbooks[]`

Relevant chunk source types:

- `audio_transcript`

Relevant textbook fields:

- `transcriptText`
- `transcriptUrl`
- `transcriptionStatus`
- `audioDuration`
- `summary`

## Current caveats

1. The controller writes `aiJobId` into textbook entries, but the textbook schema defines `videoJobId` and does not define `aiJobId`. With default Mongoose strict mode, `aiJobId` is unlikely to persist.
2. The frontend audio UI mainly relies on `transcriptionStatus` and transcript fields, so audio playback is less exposed to the `videoJobId` mismatch than video polling is.
3. Audio transcript timestamps are chunk-based, not word-level or model-native segment timestamps unless the ASR service returns them.
4. Audio shorter than 100 characters of transcript will complete without being indexed into RAG.

## Practical summary

The audio pipeline is functional and fairly robust:

- upload -> queue -> transcribe -> chunk/embed -> update subject -> summarize

Its main strengths are retry logic, chunked processing for large files, and clean RAG integration. The main architectural split to remember is that subject ingestion uses the AI job worker, while `/api/speech/*` is a separate real-time transcription surface.
